Approach 01 · The foundation

Sequoia — the
ConvNeXt baseline

80%
Kaggle · ConvNeXt-Small
The idea

Sequoia built the disciplined backbone everyone else stood on: take a strong ImageNet-22k pretrained ConvNeXt, replace its 1,000-way head with a fresh 100-way head, and fine-tune it properly with cross-validation rather than chasing tricks. A convolutional network slides learned filters across the image, building up texture → part → object features layer by layer — a great fit when you only have ~10 images per class and need every ounce of transferred knowledge.

The plan was deliberately gated: start with convnext_small, only add the heavier convnext_base and non-ConvNeXt backbones if the ensemble cleared ~0.80 out-of-fold accuracy. No wasted compute.

Data & training discipline
  • Folders 0–99 mapped with explicit label = int(folder) and an assertion that all 100 labels appear — no silent alphabetical mix-ups.
  • Every one of the 1,036 test images is predicted, not just the 1,000 rows in sample_submission.csv.
  • Stratified 5-fold CV with AdamW, layer-wise LR decay, warmup + cosine schedule, gradient clipping, label smoothing, MixUp/CutMix, and EMA checkpoint selection.
  • Out-of-fold logits saved per fold, so the ensemble decision is evidence-based, not guesswork.

Obstacles & how we tackled them
Obstacle

Alphabetical label scrambling

Reading class folders naively orders them 0, 1, 10, 11, 2… — silently mislabeling most of the dataset.

Fix

Force an integer mapping int(folder_name) and assert the labels cover exactly 0–99 before a single batch is trained.

Obstacle

The ConvNeXt-Small plateau

More epochs and a 224→288 resolution bump didn't move the needle (288px actually fell to 76% OOF). Capacity wasn't the limiter.

Fix

Diagnosed the plateau via OOF logs and pivoted strategy: the next lever is architectural diversity (a stronger ConvNeXt-Base, then non-ConvNeXt backbones) — directly setting up Kevin's and Caleb's branches.

Obstacle

Tiny dataset, constrained hardware

~10 images/class invites overfitting, and ConvNeXt-Base only fit at batch size 8 on local Apple-silicon (MPS).

Fix

Heavy regularization (label smoothing, MixUp/CutMix, EMA, drop-path) plus small batches with gradient clipping kept the larger model stable.

Reproduce on a free GPU · Kaggle T4 ×2 / Colab free
kaggle · convnext baseline

ConvNeXt-Small/Base at 224px fit comfortably in a single T4 (16 GB) with mixed precision. On Kaggle's 2×T4 you can train one backbone per GPU in parallel; on Colab free run one fold at a time and checkpoint to Drive so a dropped session resumes. 5 folds × 40 epochs lands inside Kaggle's 9-hour GPU window.

Approach 02 · Built on Sequoia's base

Kevin — squeezing
every free point

82%
Kaggle · ensemble + TTA
The idea

Kevin took Sequoia's ConvNeXt foundation and treated it as a starting line, not a finish. He ensembled ConvNeXt-Small + ConvNeXt-Base and went hunting for points the training run was leaving on the table. His baseline ensemble already cleared 81% on Kaggle — one point past Sequoia's 80% — and a found-and-fixed inference bug pushed it to 82% with zero retraining.

His next move is diversity of a different kind: bolting on a CLIP-pretrained Vision Transformer so the ensemble blends convolutional and attention-based views of the same image.

The run-by-run grind
  • Run 1 — baseline: ConvNeXt-S + ConvNeXt-B ensemble, 20 epochs, default aug, single-scale inference → 81% Kaggle.
  • Run 2 — training: doubled epochs to 40, switched to softer augmentation tuned for a small dataset, added a 2-epoch head warmup before unfreezing the backbone. OOF: ConvNeXt-S 78%, ConvNeXt-B 80.5%.
  • Run 2→3 — the free win: multi-scale TTA was configured but silently never running. Fixed it → 6 passes/image (3 scales × horizontal flip), averaged → 82% Kaggle (+1%), no retraining.
  • Run 3 — in progress: add vit_base_patch16_clip_224 (LAION-2B), weighted 3× in the ensemble. Expected 83–85%.

Obstacles & how we tackled them
Obstacle

A silent, point-leaking bug

Multi-scale test-time augmentation was fully configured — but a wiring bug meant it never actually ran. The model was scoring on single-scale inference the whole time.

Fix

Caught it by auditing the inference path, then made TTA genuinely run 6 passes per image and average them. Pure +1% for free, applied to existing checkpoints — no retraining.

Obstacle

Aggressive augmentation hurt a tiny dataset

Default CutMix/cropping was too violent for ~10 images per class — it was destroying signal faster than it regularized.

Fix

Switched to a softer augmentation profile and added a 2-epoch head warmup so the new classifier stabilized before the backbone unfroze.

Obstacle

Local hardware throughput

The data pipeline was starving the RTX 3080, leaving the GPU idle between batches.

Fix

Tuned dataloader workers for the Ryzen 7 5800X + RTX 3080 box so the GPU stayed fed through all 40 epochs.

Reproduce on a free GPU · Kaggle T4 ×2 / Colab free
kaggle · ensemble + multi-scale tta

All three backbones run at 224px and fit on a single T4 (16 GB). On Kaggle 2×T4, train ConvNeXt on one GPU and the ViT-CLIP on the other simultaneously. TTA is inference-only and cheap (6 forward passes), so it adds almost nothing to runtime. On Colab free, train sequentially and cache checkpoints — then the +1% TTA win costs only a re-run of prediction.

▸ 3 backbones · all 224px ▸ TTA = 6 passes, inference only ▸ ViT weighted ×3

python train.py --backbone vit_base_patch16_clip_224
python predict.py --tta multiscale --scales 3 --hflip --weights "1,1,3"
Approach 03 · The shipped winner

Caleb — hunting the
right backbone

87%
5-fold CV · 86.84% · DINOv2-L
The idea

Where the others perfected one backbone family, Caleb ran a bake-off. Six pretrained models — ResNet-50/101, ConvNeXt-Base, ViT-B/16, and DINOv2 Base & Large — were pushed through the same 5-fold CV pipeline over 78 hyperparameter configs. The standout was DINOv2: its self-supervised pretraining (no labels — it learns by matching a student network to a teacher's view of the same image) produces features so transferable that on ~10 images/class it crushed everything else.

Winner: DINOv2-Large, only the top 2 encoder layers unfrozen, lr 1e-4, layer-wise LR decay 0.75, 30 epochs → 86.84% cross-validated accuracy. That's the model we shipped.

Why it won — the big levers
  • Freeze depth is everything. Unfreezing 2 layers → 84.2%; the head alone → 41%; full fine-tune → 77%. On 1k images, touching more layers causes catastrophic forgetting.
  • Higher LR + LLRD beats a timid LR. 1e-4 with layer-wise decay shields the lower layers while letting the top adapt aggressively (+1.2 pts over 5e-5).
  • Augmentation hurts DINOv2. Its features are already invariant; RandAugment degraded Large by −0.9% and Base by −0.2%. MixUp (−2%) and CutMix (−6%) were measured on ConvNeXt — never tested on DINOv2 directly — and disabled preemptively.
  • Class-routed ensemble. DINOv2-Base wins 51/100 classes, Large wins 34; routing per-class picks the better expert and edges out Large alone (+0.37%).

Obstacles & how we tackled them
Obstacle

Catastrophic forgetting

Fine-tuning the whole network on 1k images wrecked the pretrained features — full fine-tune dropped to 77%, head-only stalled at 41%.

Fix

The search isolated freeze depth as the single biggest lever and landed on unfreezing just the top 2 encoder layers — 84%+ on its own.

Obstacle

A hidden layer-count bug

The freeze code hardcoded 12 layers, but DINOv2-Large has 24 — so "unfreeze top 2" was actually unfreezing the middle of the network. Two configs were silently invalid (~52%).

Fix

Added a dynamic _count_transformer_layers() that scans parameter names, so freezing targets the real top layers on any model size.

Obstacle

Generic tricks backfired on DINOv2

Progressive unfreezing quietly unfroze the entire model at epoch 3 (collapsing to 63%), and a Mixup collator leaked label-blending into validation predictions.

Fix

Disabled progressive unfreeze for DINOv2 and swapped to a plain collator before .predict(). Lesson: a robust backbone wants less intervention, not more.

Reproduce on a free GPU · Kaggle T4 ×2 / Colab free
kaggle · dinov2-large, best config

DINOv2-Large is big, but with only 2 layers unfrozen and fp16, the frozen backbone forward fits in a single T4 (16 GB) at batch 16 (drop to 8 if memory is tight). The full 78-config search is not free-tier friendly — but reproducing config 72 alone is: 5 folds × 30 epochs runs in a few hours, inside Kaggle's 9-hour GPU session. On Colab free, run one fold per session and checkpoint to Drive. On 2×T4, train Base and Large in parallel, then do the class-routed merge.

Where we are now —
and what's next.

We compared every approach and shipped the highest-accuracy model — Caleb's DINOv2-Large at 86.84%. But we're treating it as a checkpoint, not a trophy. All three lines of work are still live, and we keep trading findings to push the number higher before the deadline.

The race so far

Kevin tunes it up · 81% → 82%

Caleb finds the winner · 86.84%

Now · still climbing

What we're trying next
01 · resolution

DINOv2 at native 518px

DINOv2 was pretrained at 518px but we run it at 224px, throwing away spatial detail. Testing the full-resolution input next.

02 · cross-pollinate

One unified ensemble

Blend Caleb's DINOv2 with Kevin's CLIP-ViT and Sequoia's ConvNeXt — convolution + attention + self-supervised, all voting together.

03 · the data ceiling

Semi-supervised pseudo-labeling

At ~10 images/class we're near the data ceiling; pseudo-labeling the test set or borrowing similar public data is the likely next gain.

04 · hard classes

Rescue class #66 and friends

A handful of classes score near zero F1 across every model. Targeted error analysis and per-class routing are on the list.
